● 现在我已经掌握了完整的业务链路。以下是分析：

  ---
  回复评论 → 发送邮件的业务链路

  调用链

  POST /comment (CommentController.postComment)
    │
    ├─ 判断身份：admin 还是 visitor
    │    ├─ admin → setAdminComment() → notice=false, adminComment=true
    │    └─ visitor → setVisitorComment() → adminComment=false, notice 未设置!
    │
    ├─ commentService.saveComment(comment)    // 先保存到数据库
    │
    └─ commentUtils.judgeSendNotify(comment, isVisitorComment, parentComment)
         │
         ├─ [路径A] 通知被回复的人（父评论者）
         │    条件: parentComment != null
         │         && !parentComment.getAdminComment()   // 父评论不是博主的
         │         && parentComment.getNotice()           // 父评论者开启了提醒
         │    → sendMailToParentComment() → mailUtils.sendHtmlTemplateMail(...,
  "guest.html")
         │
         └─ [路径B] 通知博主自己
             条件: isVisitorComment == true  // 只有访客评论才触发
             → notifyMyself() → MailChannel.notifyMyself() →
  mailUtils.sendHtmlTemplateMail(..., "owner.html")

  管理员回复访客的场景（你说的"回复评论"）

  当管理员回复访客评论时：
  - isVisitorComment = false → 路径B 不会触发（合理，管理员不需要通知自己）
  - 只有 路径A 可能触发：给被回复的访客发邮件
  - 路径A 的条件是：parentComment.getNotice() == true

  ---
  根因分析：问题出在代码逻辑，不是配置

  关键在 CommentUtils.java:278 的 setVisitorComment() 方法：

  public void setVisitorComment(Comment comment, HttpServletRequest request) {
      comment.setNickname(comment.getNickname().trim());
      setCommentRandomAvatar(comment);
      // ...
      comment.setAdminComment(false);
      comment.setCreateTime(new Date());
      comment.setPublished(commentDefaultOpen);
      comment.setEmail(comment.getEmail().trim());
      comment.setIp(IpAddressUtils.getIpAddress(request));
      // ⚠️ notice 没有被设置！既没有设为 true，也没有从配置读取
  }

  对比 setAdminComment() 明确设置了 comment.setNotice(false)，而访客评论的
  notice 完全没有被初始化。

  导致两个问题：

  1. notice 完全依赖前端传值。如果前端评论表单没有传 notice: 
  true（比如根本没有"回复时邮件通知我"这个复选框，或复选框未勾选），那么 notice
  就是 null/false。当这个评论被保存到数据库后，管理员回复它时，parentComment.get
  Notice() 返回 null 或 false，路径A的条件不满足，邮件不会发送。
  2. 如果 notice 为 null，存在 NPE 风险。parentComment.getNotice() 返回
  Boolean(null)，在 && 判断中自动拆箱为 boolean 时会抛出
  NullPointerException。这会导致评论已保存但接口返回 500 错误。

  另一个隐患：@EnableAsync 位置不当

  MailUtils.java:22 把 @EnableAsync 放在 @Component 类上而非 @Configuration
  类上，这是非标准用法，在某些 Spring Boot 版本中可能导致 @Async 不生效。

  还有：邮件发送异常被静默吞掉

  MailUtils.java:59-61 的 catch (Exception e) { e.printStackTrace(); }
  会静默吞掉所有 SMTP 异常。即使代码走到了发送逻辑，如果 SMTP
  配置有问题（密码错误、服务器不通等），也不会报错。

  ---
  修复建议

  最直接的修复是在 setVisitorComment() 中给 notice 一个默认值：

  // 在 setVisitorComment() 中添加
  comment.setNotice(true);  // 默认开启邮件提醒

  或者通过配置控制：

  @Value("${comment.notice-default}")
  private Boolean noticeDefault;

  // setVisitorComment() 中:
  comment.setNotice(noticeDefault);

  同时建议把 @EnableAsync 移到独立的 @Configuration 类上，确保异步机制可靠生效。

  需要我帮你修复这些问题吗？
